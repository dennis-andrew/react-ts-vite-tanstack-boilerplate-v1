# TanStack Router Implementation and Migration Guide

## Document Purpose

This document defines the routing architecture, implementation conventions,
and migration strategy for TanStack Router in this repository. It is intended
for engineers who are familiar with React and React Router and require a
repository-specific reference for implementing, reviewing, or extending route
behavior.

The document has four objectives:

1. Describe the routing architecture currently implemented in the repository.
2. Establish conventions for adding and maintaining routes.
3. Map React Router concepts to their TanStack Router equivalents.
4. Provide implementation patterns for routing features expected in a
   production application.

The repository uses the following relevant technologies:

- React 19
- TanStack Router 1.x
- TanStack Query 5
- TypeScript in strict mode
- File-based routing generated by the TanStack Router Vite plugin
- A client-side Vite single-page application

All examples are aligned with the dependency versions declared in
`package.json`. TanStack Router has an active release cycle; proposed use of an
API not represented by the installed package types should be validated against
the official documentation and the repository's locked dependency versions.

### Scope

This guide covers client-side routing in the existing Vite single-page
application. It does not define a server-side rendering architecture, a
TanStack Start migration, or a general-purpose routing standard for unrelated
repositories.

### Normative terminology

The terms **must**, **must not**, **should**, and **should not** indicate
repository conventions:

- **Must / must not** identifies a requirement necessary for correctness,
  security, type safety, or consistency.
- **Should / should not** identifies the preferred implementation unless a
  documented exception applies.

## 1. Standard Procedure for Adding a Protected Route

The following procedure adds a protected `/reports` route:

1. Create `src/views/Reports/index.tsx`.
2. Create `src/routes/app/_authenticated.reports.tsx`.
3. Export a `Route` created with `createFileRoute`.
4. Add `/reports` to `AppRoutes` if shared navigation or redirects use it.
5. Run `pnpm build` or restart `pnpm dev`.
6. Do not edit `src/routeTree.gen.ts`; it is generated by the router plugin.

```tsx
// src/routes/app/_authenticated.reports.tsx
import { createFileRoute } from '@tanstack/react-router'
import Reports from 'src/views/Reports'

export const Route = createFileRoute('/_authenticated/reports')({
  component: Reports,
})
```

The internal route ID includes `/_authenticated`, while the browser URL is
`/reports`. The leading underscore defines `_authenticated` as a pathless
layout. Its `beforeLoad` guard applies to all descendant routes.

## 2. Architectural Model

React Router commonly represents route configuration as JSX:

```tsx
<Routes>
  <Route element={<ProtectedLayout />}>
    <Route path="/reports" element={<Reports />} />
  </Route>
</Routes>
```

In this boilerplate, the file system is the route configuration:

```text
src/routes/app/
├── __root.tsx
├── _authenticated.tsx
├── _authenticated.index.tsx
├── _authenticated.reports.tsx
├── auth.tsx
└── auth.login.tsx
```

The Vite plugin reads these files and generates `src/routeTree.gen.ts`. The
generated route tree provides TypeScript with the application's route IDs,
paths, path parameters, search parameters, loader results, and router-context
types.

A TanStack route is an application boundary rather than only a
URL-to-component mapping. A route may define:

- URL matching
- Path-parameter parsing
- Search-parameter validation
- Authentication and authorization
- Data dependencies
- Data loading
- Loading, error, and not-found UI
- Preloading and cache timing
- Document metadata
- Lifecycle behavior

### Route loading order

For a matched route branch, the relevant execution order is:

1. Match route files against the URL.
2. Parse path params.
3. Validate search params.
4. Run parent `beforeLoad` functions from top to bottom.
5. Run route loaders, normally in parallel once their guards allow them.
6. Render nested components through `<Outlet />`.

The execution order has the following architectural implications:

- Authentication belongs in `beforeLoad`.
- URL search values must be validated before loaders use them.
- A parent pathless route can protect all descendants.
- A redirect thrown by a parent prevents child loaders and components from
  running.

## 3. Repository Integration

### Vite plugin

`vite.config.ts` contains:

```ts
tanstackRouter({
  target: 'react',
  routesDirectory: './src/routes/app',
  generatedRouteTree: './src/routeTree.gen.ts',
  routeFileIgnorePattern: '\\.(test|spec)\\.[tj]sx?$',
  autoCodeSplitting: true,
})
```

The configuration establishes the following behavior:

- Route modules live in `src/routes/app`.
- The generated tree is written to `src/routeTree.gen.ts`.
- Route-adjacent tests are ignored as route files.
- Route code is automatically split into lazy chunks where possible.
- The plugin must run before the React Vite plugin.

### Root route

`src/routes/app/__root.tsx` defines the root of the route tree:

```tsx
export const Route = createRootRouteWithContext<RouterContext>()({
  component: RootRoute,
})

function RootRoute() {
  return <Outlet />
}
```

`createRootRouteWithContext<RouterContext>()` establishes the router-context
type for every descendant.

### Router instance

`src/routes/index.tsx`:

- Imports the generated `routeTree`.
- Creates the router once.
- Registers its type globally.
- Supplies global loading, error, and not-found components.
- Enables intent preloading.
- Passes authentication and Query Client through router context.
- Invalidates routes when auth state changes.
- Renders Router Devtools only in development.

The global type registration is required:

```ts
declare module '@tanstack/react-router' {
  interface Register {
    router: typeof router
  }
}
```

Without this registration, `<Link>`, `useNavigate`, and related APIs cannot
infer the application's valid route contracts.

### Provider hierarchy

The effective hierarchy is:

```text
React.StrictMode
└── RequireNetwork
    └── AuthProvider
        └── AppRouter
            └── QueryClientProvider
                └── RouterProvider
```

`AppRouter` reads `AuthContext` and passes that value into `RouterProvider`.
Route guards do not call React hooks; they read `context.auth`.

### Existing route tree

| Route file                 | Route ID           | Browser URL       | Purpose                      |
| -------------------------- | ------------------ | ----------------- | ---------------------------- |
| `__root.tsx`               | `__root__`         | all URLs          | Root outlet and context type |
| `_authenticated.tsx`       | `/_authenticated`  | no segment        | Protected pathless layout    |
| `_authenticated.index.tsx` | `/_authenticated/` | `/`               | Protected home page          |
| `auth.tsx`                 | `/auth`            | `/auth`           | Auth layout                  |
| `auth.login.tsx`           | `/auth/login`      | `/auth/login`     | Login page                   |
| `app-components.tsx`       | `/app-components`  | `/app-components` | Development-only showcase    |

Route IDs and browser URLs are related but are not necessarily identical.
Pathless layout IDs retain the underscore-prefixed segment; browser URLs omit
that segment.

## 4. React Router to TanStack Router Translation

| React Router                | TanStack Router in this repository                            |
| --------------------------- | ------------------------------------------------------------- |
| `<BrowserRouter>`           | `createRouter` + `<RouterProvider>`                           |
| `<Routes>` and `<Route>`    | Files in `src/routes/app`                                     |
| Route `element`             | Route `component`                                             |
| `<Outlet />`                | `<Outlet />` from `@tanstack/react-router`                    |
| `<Link to="/users/1">`      | `<Link to="/users/$userId" params={{ userId: '1' }}>`         |
| `useNavigate()`             | `useNavigate()` with typed navigation objects                 |
| `navigate('/users/1')`      | `navigate({ to: '/users/$userId', params: { userId: '1' } })` |
| `<Navigate to="/login" />`  | `<Navigate to="/login" />`                                    |
| `useParams()`               | `Route.useParams()` or typed global `useParams({ from })`     |
| `useSearchParams()`         | `validateSearch` + `Route.useSearch()` + navigation `search`  |
| `useLocation()`             | `useLocation()`                                               |
| `useMatches()`              | `useMatches()`                                                |
| `loader`                    | Route `loader`                                                |
| `action`                    | TanStack Query mutation under the repository architecture     |
| `errorElement`              | `errorComponent`                                              |
| route redirect              | `throw redirect(...)` in `beforeLoad` or `loader`             |
| protected wrapper component | Pathless route with `beforeLoad`                              |
| `useBlocker`                | TanStack Router `useBlocker`                                  |
| lazy route element          | Automatic route code splitting                                |

### Principal migration difference

React Router implementations frequently construct dynamic URLs as strings:

```ts
navigate(`/users/${userId}?tab=activity`)
```

TanStack Router navigation should use a structured navigation object:

```ts
navigate({
  to: '/users/$userId',
  params: { userId },
  search: { tab: 'activity' },
})
```

Structured navigation allows TypeScript to validate route paths, path
parameters, and search parameters during implementation and refactoring.

## 5. File Naming Rules

This boilerplate currently uses flat route filenames. Dots represent nesting.

| Filename                      | Meaning                                               |
| ----------------------------- | ----------------------------------------------------- |
| `__root.tsx`                  | Required root route                                   |
| `about.tsx`                   | `/about`                                              |
| `posts.tsx`                   | Parent route `/posts`                                 |
| `posts.index.tsx`             | Exact index page `/posts`                             |
| `posts.$postId.tsx`           | Dynamic page `/posts/:postId`                         |
| `posts.$postId.edit.tsx`      | `/posts/:postId/edit`                                 |
| `files.$.tsx`                 | Catch-all under `/files`                              |
| `_authenticated.tsx`          | Pathless layout; `_authenticated` is omitted from URL |
| `_authenticated.settings.tsx` | Protected `/settings`                                 |
| `posts_.tsx`                  | Escapes nesting under a matching `posts` parent       |
| `-helpers.ts`                 | Ignored by the route generator                        |

Important symbols:

- `.` creates route nesting in flat filenames.
- `$name` creates a required path param.
- `$` by itself creates a wildcard/splat param named `_splat`.
- `_name` creates a pathless layout.
- `name_` escapes a route from an otherwise matching parent.
- `index` creates an index route.
- `-name` excludes a file or folder from route generation.

Directory-style routes are also supported, but use one convention consistently
within a feature. For example:

```text
src/routes/app/projects/
├── route.tsx
├── index.tsx
└── $projectId.tsx
```

The current repository uses flat route files. New route modules should retain
this convention unless a broader route-folder migration is approved.

### Shell handling for `$`

Shells treat `$name` as a variable. Quote dynamic route filenames:

```bash
touch 'src/routes/app/_authenticated.users.$userId.tsx'
```

## 6. Adding Routes

### Public page

```tsx
// src/routes/app/about.tsx
import { createFileRoute } from '@tanstack/react-router'
import About from 'src/views/About'

export const Route = createFileRoute('/about')({
  component: About,
})
```

### Protected page

```tsx
// src/routes/app/_authenticated.settings.tsx
import { createFileRoute } from '@tanstack/react-router'
import Settings from 'src/views/Settings'

export const Route = createFileRoute('/_authenticated/settings')({
  component: Settings,
})
```

Browser URL: `/settings`.

### Nested layout

```tsx
// src/routes/app/_authenticated.projects.tsx
import { createFileRoute, Outlet } from '@tanstack/react-router'
import ProjectsSidebar from 'src/views/Projects/ProjectsSidebar'

export const Route = createFileRoute('/_authenticated/projects')({
  component: ProjectsLayout,
})

function ProjectsLayout() {
  return (
    <div>
      <ProjectsSidebar />
      <main>
        <Outlet />
      </main>
    </div>
  )
}
```

```tsx
// src/routes/app/_authenticated.projects.index.tsx
export const Route = createFileRoute('/_authenticated/projects/')({
  component: ProjectsList,
})
```

```tsx
// src/routes/app/_authenticated.projects.$projectId.tsx
export const Route = createFileRoute('/_authenticated/projects/$projectId')({
  component: ProjectDetails,
})
```

The parent layout remains mounted while its child outlet changes.

### Index route versus parent route

`projects.tsx` matches the layout branch. `projects.index.tsx` provides the
content for the exact `/projects` URL. Without an index child, the parent's
`<Outlet />` is empty at `/projects`.

### Catch-all route

```tsx
// src/routes/app/docs.$.tsx
export const Route = createFileRoute('/docs/$')({
  component: DocsPage,
})

function DocsPage() {
  const { _splat } = Route.useParams()
  return <div>Requested docs path: {_splat}</div>
}
```

### Optional path param

TanStack Router 1.x supports optional path params using `{-$name}`:

```tsx
// Route path: /articles/{-$category}
export const Route = createFileRoute('/articles/{-$category}')({
  component: Articles,
})

function Articles() {
  const { category } = Route.useParams()
  return <div>{category ?? 'All categories'}</div>
}
```

Prefer search params for optional filters. Use optional path params when the
path itself expresses resource hierarchy.

## 7. Navigation

### Declarative links

Use `<Link>` for navigational elements that users can activate, copy, inspect,
or open in a separate browser context.

```tsx
import { Link } from '@tanstack/react-router'

<Link to="/settings">Settings</Link>

<Link to="/users/$userId" params={{ userId: user.id }}>
  View user
</Link>
```

Internal dynamic URLs should not be assembled manually when a typed `to` and
`params` combination is available.

### Active links

```tsx
<Link
  to="/settings"
  activeProps={{ className: 'navigation-link--active' }}
  inactiveProps={{ className: 'navigation-link' }}
>
  Settings
</Link>
```

Use `activeOptions={{ exact: true }}` when a link should only be active for the
exact destination rather than descendants.

### Imperative navigation

Use imperative navigation after events such as successful form submissions:

```tsx
const navigate = useNavigate()

await saveUser(values)

await navigate({
  to: '/users/$userId',
  params: { userId: values.id },
  replace: true,
})
```

Inside a route component, prefer its bound hook:

```tsx
const navigate = Route.useNavigate()
```

This automatically scopes relative navigation from that route.

### Relative navigation

```tsx
<Link to=".">Current route</Link>
<Link to="..">Parent route</Link>
```

Explicit absolute `to` paths are generally preferable in reusable shared
components because they make the navigation contract unambiguous.

### Navigate component

For a client-only render-time redirect:

```tsx
<Navigate to="/settings" replace />
```

For access rules and route decisions, prefer `throw redirect(...)` from
`beforeLoad` so the protected component never renders.

### Hash, history state, and replace

```ts
navigate({
  to: '/reports',
  hash: 'totals',
  state: { openedFrom: 'dashboard' },
  replace: true,
})
```

- `hash` targets an in-page fragment.
- `state` is non-shareable history state; it disappears when a URL is copied.
- `replace` replaces the current history entry instead of adding one.

Use search params instead of history state for data that must survive refresh,
bookmarking, or sharing.

### External URLs

Use a normal anchor for external navigation:

```tsx
<a href="https://example.com" target="_blank" rel="noreferrer">
  External documentation
</a>
```

The repository's `useRedirect` hook supports `href` because login redirect
targets are validated as internal URLs. Untrusted external input must not be
passed to this API.

### Back-navigation with a fallback

The shared hook exposes:

```ts
const { navigateBack } = useRedirect()
navigateBack()
```

For a button that needs a fallback:

```tsx
import { useCanGoBack, useNavigate, useRouter } from '@tanstack/react-router'

const canGoBack = useCanGoBack()
const router = useRouter()
const navigate = useNavigate()

const goBack = () => {
  if (canGoBack) {
    router.history.back()
  } else {
    navigate({ to: '/' })
  }
}
```

## 8. Path Params

### Define and read a param

```tsx
// src/routes/app/_authenticated.users.$userId.tsx
export const Route = createFileRoute('/_authenticated/users/$userId')({
  component: UserPage,
})

function UserPage() {
  const { userId } = Route.useParams()
  return <div>User {userId}</div>
}
```

`Route.useParams()` is preferred inside a route module because it is strict and
fully typed.

Outside the route module:

```tsx
import { useParams } from '@tanstack/react-router'

const { userId } = useParams({
  from: '/_authenticated/users/$userId',
})
```

Use `strict: false` only in route-agnostic shared components. This option
weakens type precision because the component may render under multiple route
branches.

### Parse params into domain types

Path params arrive as strings by default. A route can parse and stringify them:

```tsx
export const Route = createFileRoute('/orders/$orderId')({
  params: {
    parse: ({ orderId }) => {
      if (!/^\d+$/.test(orderId)) return false
      return { orderId: Number(orderId) }
    },
    stringify: ({ orderId }) => ({
      orderId: String(orderId),
    }),
  },
  component: OrderPage,
})
```

Now `Route.useParams().orderId` is a number, and navigation requires a number.
Returning `false` lets the matcher try another compatible route.

Param parsing should be introduced only when the domain type provides a
material correctness benefit. Retaining identifiers as strings is generally
simpler and preserves leading zeroes.

## 9. Search Params: Typed URL State

Search params should hold shareable UI state:

- Pagination
- Search terms
- Filters
- Sort order
- Selected tabs
- Optional view modes

Raw search input must be treated as untrusted because URLs can be edited
directly or supplied by an external source.

### Validate search params

This repository supplies validation helpers in
`src/routes/routerSearchUtils.ts`:

```tsx
const statuses = ['all', 'active', 'archived'] as const

export const Route = createFileRoute('/_authenticated/users')({
  validateSearch: (search) => ({
    page: readPositiveIntegerSearchParam(search.page, 1),
    query: readTrimmedStringSearchParam(search.query),
    status: readEnumSearchParam(search.status, statuses, 'all'),
    compact: readBooleanSearchParam(search.compact),
  }),
  component: UsersPage,
})
```

The validator is both a runtime boundary and a TypeScript type source.

### Read search params

```tsx
function UsersPage() {
  const { page, query, status, compact } = Route.useSearch()
  // page is number, status is a union, compact is boolean
}
```

Outside the route module:

```tsx
const search = useSearch({ from: '/_authenticated/users' })
```

### Update search params

```tsx
const navigate = Route.useNavigate()

const setPage = (page: number) => {
  navigate({
    search: (previous) => ({
      ...previous,
      page,
    }),
  })
}
```

Filter changes should generally reset pagination:

```tsx
navigate({
  search: (previous) => ({
    ...previous,
    query: nextQuery,
    page: 1,
  }),
  replace: true,
})
```

Functional updates preserve unrelated search values inherited from parent
routes.

### Search params in links

```tsx
<Link
  to="/users"
  search={{
    page: 1,
    query: '',
    status: 'active',
    compact: false,
  }}
>
  Active users
</Link>
```

### Search inheritance

Search params validated by a parent are inherited by descendants. Avoid putting
page-specific search validation on `__root.tsx`; every route would inherit it.

Root- or layout-level search state should be used only when the complete route
branch owns the value, such as a workspace selection shared across all
workspace pages.

### Safe login redirects

`src/routes/redirectSearch.ts` rejects:

- External absolute URLs such as `https://attacker.example`
- Protocol-relative URLs such as `//attacker.example`
- Non-string values

It accepts internal paths beginning with one `/`. This prevents an open
redirect vulnerability.

Any route that accepts a return URL should reuse this validator.

## 10. Authentication and Authorization

### Existing protected layout

`src/routes/app/_authenticated.tsx` checks:

- Access token
- Refresh token
- `context.auth.authenticated`

If any check fails, it throws:

```ts
throw redirect({
  to: AppRoutes.AUTH.LOGIN,
  search: {
    redirect: location.href,
  },
  replace: true,
})
```

The attempted URL is preserved, including search and hash.

### Why `beforeLoad` is correct

A component-level check briefly renders or begins loading the protected route.
A parent `beforeLoad` runs before descendant loaders and components.

Use:

- `beforeLoad` for authentication and authorization.
- `loader` for fetching route-critical data.
- Components for presentation and interaction.

### Role-based authorization

Create another pathless layout beneath `_authenticated`:

```tsx
// src/routes/app/_authenticated._admin.tsx
import { createFileRoute, Outlet, redirect } from '@tanstack/react-router'

export const Route = createFileRoute('/_authenticated/_admin')({
  beforeLoad: ({ context }) => {
    if (context.auth.user?.role !== 'admin') {
      throw redirect({ to: '/', replace: true })
    }
  },
  component: () => <Outlet />,
})
```

```tsx
// src/routes/app/_authenticated._admin.audit.tsx
export const Route = createFileRoute('/_authenticated/_admin/audit')({
  component: AuditPage,
})
```

Browser URL: `/audit`.

For denied access, choose deliberately between:

- Redirecting to a safe page.
- Throwing `notFound()` to hide the resource's existence.
- Throwing a typed error and rendering a 403-specific error component.

### Auth context invalidation

`AppRouter` calls `router.invalidate()` when authentication or the user changes.
This causes route guards and loaders that depend on router context to be
re-evaluated.

`router.invalidate()` should not be called for every server mutation. Mutations
should invalidate the relevant TanStack Query keys. Router invalidation is
appropriate only when route-level decisions, context dependencies, or loaders
require re-evaluation.

## 11. Data Loading

### Plain route loader

```tsx
export const Route = createFileRoute('/_authenticated/users/$userId')({
  loader: async ({ params }) => {
    const user = await getUser(params.userId)
    if (!user) throw notFound()
    return user
  },
  component: UserPage,
})

function UserPage() {
  const user = Route.useLoaderData()
  return <h1>{user.name}</h1>
}
```

Use a route loader when data:

- Is required before the page makes sense.
- Can redirect.
- Can produce route-level not-found behavior.
- Should preload when a user shows intent to visit.

### Search-driven loader dependencies

Declare exactly which search values cause a loader to reload:

```tsx
export const Route = createFileRoute('/_authenticated/users')({
  validateSearch: (search) => ({
    page: readPositiveIntegerSearchParam(search.page, 1),
    query: readTrimmedStringSearchParam(search.query),
  }),
  loaderDeps: ({ search }) => ({
    page: search.page,
    query: search.query,
  }),
  loader: ({ deps }) => getUsers(deps),
  component: UsersPage,
})
```

Loader logic must not depend on search params that are omitted from
`loaderDeps`. The dependency declaration allows the router to determine when a
loader result is stale.

### TanStack Query integration used by this repository

The shared `queryClient` is available as `context.queryClient`:

```tsx
const userQueryOptions = (userId: string) =>
  queryOptions({
    queryKey: ['users', 'detail', userId],
    queryFn: () => getUser(userId),
  })

export const Route = createFileRoute('/_authenticated/users/$userId')({
  loader: ({ context, params }) =>
    context.queryClient.ensureQueryData(userQueryOptions(params.userId)),
  component: UserPage,
})

function UserPage() {
  const { userId } = Route.useParams()
  const { data: user } = useSuspenseQuery(userQueryOptions(userId))
  return <h1>{user.name}</h1>
}
```

This pattern establishes a single server-state source of truth:

- The loader ensures data exists and participates in route preloading.
- The component subscribes to the Query cache.
- Query owns server cache, retries, invalidation, and background refetching.

Keep query option factories in feature service files so routes, components,
prefetching code, and mutations use identical query keys.

### Router cache versus Query cache

TanStack Router has its own loader cache, but this project already uses
TanStack Query for server state. Prefer Query as the server-data cache and use
route loaders as orchestration.

Avoid storing the same API response independently in:

- Route loader cache
- Query cache
- React Context
- Component state

### Mutations

Use Query mutations for create, update, and delete operations:

```tsx
const mutation = useMutation({
  mutationFn: updateUser,
  onSuccess: async (user) => {
    await queryClient.invalidateQueries({ queryKey: ['users'] })
    await navigate({
      to: '/users/$userId',
      params: { userId: user.id },
    })
  },
})
```

TanStack Router does not require React Router-style route actions in this
client-side architecture.

## 12. Pending, Error, and Not-Found States

### Global defaults

`src/routes/index.tsx` configures:

```ts
defaultPendingComponent: RoutePendingFallback,
defaultErrorComponent: RouteErrorFallback,
defaultNotFoundComponent: RouteNotFoundFallback,
```

These components provide the application-wide fallback behavior.

### Route-specific states

```tsx
export const Route = createFileRoute('/reports')({
  loader: loadReports,
  pendingComponent: ReportsSkeleton,
  errorComponent: ReportsError,
  notFoundComponent: ReportsNotFound,
  component: ReportsPage,
})
```

Use route-specific UI when the user can recover in a feature-specific way.

### Throwing not found

```ts
const project = await getProject(params.projectId)
if (!project) throw notFound()
```

Use `notFound()` for a missing route resource, not for ordinary empty lists.
An empty search result is valid page data and should render an empty state.

### Throwing redirects

```ts
throw redirect({
  to: '/projects/$projectId',
  params: { projectId },
  replace: true,
})
```

Redirects should be thrown from guards and loaders and must not be handled as
ordinary failures. If a broad `try/catch` is required, recognized redirect and
not-found objects must be rethrown.

### Error reset

An error component receives route error information and reset behavior. It
should provide a retry action when reloading the route can recover. Raw server
stack traces and sensitive error details must not be exposed in user-facing
output.

## 13. Preloading and Code Splitting

The router uses:

```ts
defaultPreload: 'intent'
```

Hovering or focusing a `<Link>` can preload the destination route. Depending on
the route configuration, preloading may load:

- The lazy route chunk
- Route guards
- Route loaders
- Query data ensured by the loader

This reduces navigation latency without preloading the complete application.

Override per link when needed:

```tsx
<Link to="/heavy-report" preload={false}>
  Heavy report
</Link>
```

Automatic code splitting is enabled in `vite.config.ts`. Keep route files thin
and import page components normally. Confirm chunk behavior with a production
build instead of manually adding `React.lazy` to every route.

Run:

```bash
pnpm build
```

Then inspect the generated `build/assets` chunks or use browser network tools.

## 14. Router Context

Router context is for dependencies needed before rendering:

```ts
export interface RouterContext {
  auth: RouterAuthContext
  queryClient: QueryClient
}
```

Appropriate router-context values:

- Authentication/session state
- Query Client
- Feature-flag client used by guards
- Dependency-injected API clients

Values that should remain outside router context:

- Page form state
- Modal visibility
- Large changing view models
- Data already owned by TanStack Query

Add a context value in three places:

1. `RouterContext` type.
2. Placeholder `context` passed to `createRouter`.
3. Real `context` passed to `<RouterProvider>`.

If route decisions depend on a changed context value, call
`router.invalidate()`.

### Extending context in a route

A parent `beforeLoad` may return context for descendants:

```tsx
export const Route = createFileRoute('/_authenticated/workspaces/$workspaceId')(
  {
    beforeLoad: async ({ params }) => {
      const workspace = await getWorkspace(params.workspaceId)
      if (!workspace) throw notFound()
      return { workspace }
    },
    component: () => <Outlet />,
  },
)
```

Child loaders can then read `context.workspace`. This is useful for a resolved
parent resource or permissions shared across a branch.

## 15. Reading Router State and Matching Routes

### Current location

```tsx
const location = useLocation()

location.pathname
location.search
location.hash
location.href
```

Select only what a component needs to reduce rerenders:

```tsx
const pathname = useRouterState({
  select: (state) => state.location.pathname,
})
```

### Match a route

```tsx
const matchRoute = useMatchRoute()
const userParams = matchRoute({ to: '/users/$userId' })

if (userParams) {
  // The route is active and userParams contains matched params.
}
```

Use this for navigation state or branch-aware UI. Prefer `<Link>` active props
for ordinary navigation links.

### All matched routes

```tsx
const matches = useMatches()
```

This is useful for breadcrumbs and route metadata.

### Static route data for breadcrumbs

For a larger application, route static-data types may be registered and
breadcrumb labels attached to route definitions. Breadcrumbs can then be
derived from `useMatches()`. Dynamic labels, such as a user's name, should
remain in loader data.

The existing `BreadCrumb` component currently receives breadcrumb data through
props. Introduce match-driven breadcrumbs only if the application needs a
centralized route metadata system.

## 16. Scroll, Hashes, and History

TanStack Router supports scroll restoration, hash scrolling, and multiple
history implementations.

This browser SPA uses the default browser history. Tests use:

```ts
createMemoryHistory({
  initialEntries: ['/'],
})
```

Use memory history in tests and non-browser environments.

For hash navigation:

```tsx
<Link to="/reports" hash="totals">
  Report totals
</Link>
```

When enabling custom scroll restoration, configure it once at router creation
and test browser back/forward behavior. Avoid feature components each inventing
their own global scroll policy.

## 17. Blocking Navigation for Unsaved Changes

Use `useBlocker` for a dirty form:

```tsx
const { status, proceed, reset } = useBlocker({
  shouldBlockFn: () => formState.isDirty,
  enableBeforeUnload: formState.isDirty,
  withResolver: true,
})

return (
  <>
    <EditForm />
    {status === 'blocked' && (
      <ConfirmLeaveModal onLeave={proceed} onStay={reset} />
    )}
  </>
)
```

- `shouldBlockFn` decides whether in-app navigation is blocked.
- `enableBeforeUnload` handles reload, tab close, and browser exit.
- `withResolver` lets custom UI call `proceed` or `reset`.

Navigation should be blocked only when leaving the route would discard unsaved
user work. The blocking condition must be cleared after a successful save.

## 18. Document Titles and Metadata

TanStack Router can define `head` metadata per route. In a Vite SPA,
`<HeadContent />` must be rendered near the top of the root route:

```tsx
import {
  createRootRouteWithContext,
  HeadContent,
  Outlet,
} from '@tanstack/react-router'

function RootRoute() {
  return (
    <>
      <HeadContent />
      <Outlet />
    </>
  )
}
```

Then a route can provide metadata:

```tsx
export const Route = createFileRoute('/reports')({
  head: () => ({
    meta: [
      { title: 'Reports' },
      { name: 'description', content: 'View application reports' },
    ],
  }),
  component: ReportsPage,
})
```

Before enabling this capability in the repository:

1. Remove or deliberately retain the static `<title>` in `index.html`.
2. Add `<HeadContent />` to `__root.tsx`.
3. Define a root fallback title.
4. Verify title changes during client navigation.

Nested route metadata overrides matching parent title/meta entries.

## 19. Route Masking

Route masking displays one URL while rendering another route. A common use is a
modal that has a full-page fallback:

```tsx
<Link
  to="/photos/$photoId/modal"
  params={{ photoId }}
  mask={{
    to: '/photos/$photoId',
    params: { photoId },
  }}
>
  Open photo
</Link>
```

The app renders the modal route while the address bar shows the masked photo
URL. If the URL is shared outside the current history state, it behaves as the
normal unmasked destination.

Route masking should be limited to interaction patterns that require distinct
rendered and displayed locations, such as modal-detail navigation. It adds
additional URL semantics and corresponding test requirements.

## 20. Router Events and Route Lifecycle

Global analytics and diagnostics should subscribe to router events from the
router instance rather than duplicate tracking effects across page components.

For route-specific behavior, TanStack Router supports lifecycle options for
entering, retaining, and leaving a match. React effects remain appropriate for
UI subscriptions owned by a rendered component. Router lifecycle behavior
should be used only when the concern is directly associated with route
matching.

Always unsubscribe global event listeners during cleanup.

## 21. Route Testing in This Repository

### Router-aware component tests

`customRender` uses a memory router by default:

```tsx
const { user, router } = customRender(<Pagination />, {
  initialEntries: ['/users?page=2'],
})
```

After interaction:

```tsx
await user.click(screen.getByRole('button', { name: 'Next' }))

expect(router?.state.location.search).toMatchObject({
  page: 3,
})
```

Use `withRouter: false` only when the component has no router hooks, links, or
router state.

### Test a guard directly

```tsx
const redirectError = expectRedirect(() => {
  Route.options.beforeLoad?.({
    context: { auth: { authenticated: false } },
    location: { href: '/reports?page=2' },
  } as never)
})

expect(redirectError.options).toMatchObject({
  to: '/auth/login',
  search: { redirect: '/reports?page=2' },
})
```

Direct guard tests provide focused verification of access-control rules.

### Test search validation

```tsx
const validateSearch = Route.options.validateSearch as (
  input: Record<string, unknown>,
) => UsersSearch

expect(validateSearch({ page: '-5', status: 'invalid' })).toEqual({
  page: 1,
  status: 'all',
})
```

### Test loaders

Test loaders directly when they contain:

- Branching
- Data transformation
- Redirect behavior
- Not-found behavior
- Query-key construction

Use `expectNotFound()` for missing-resource branches.

### Integration test a real route tree

For navigation behavior involving nested layouts, guards, or loader rendering,
create a memory router with the real generated `routeTree`, pass test context,
render `RouterProvider`, and await `router.load()`.

Tests should use the smallest scope that verifies the required behavior. A full
route-tree test has higher setup and execution cost and requires representative
authentication and Query context.

## 22. Migration Strategy from React Router

Migration into this boilerplate should proceed in the following sequence:

### Step 1: Inventory existing routing

Search for:

```bash
rg "react-router|react-router-dom|useSearchParams|useParams|useNavigate|<Route|<Routes|<Outlet"
```

List:

- All paths
- Layout nesting
- Protected branches
- Dynamic params
- Search params and defaults
- Redirects
- Loaders/actions
- Error boundaries
- Navigation blockers

### Step 2: Define the route hierarchy

Define the logical route hierarchy before selecting filenames:

- Root layout
- Public layouts
- Protected pathless layouts
- URL-owning layouts
- Index routes
- Dynamic and catch-all routes

Translate the approved hierarchy into route files after layouts, ownership, and
access boundaries are established.

### Step 3: Create route modules

Create one route file per route. Keep existing page components in `src/views`
and import them into thin route modules.

### Step 4: Move protection into `beforeLoad`

Replace protected wrapper components and component-level redirect effects with
pathless guarded layouts.

### Step 5: Migrate params

Replace:

```tsx
const { userId } = useParams()
```

with:

```tsx
const { userId } = Route.useParams()
```

Replace interpolated navigation strings with typed `params`.

### Step 6: Migrate search state

For each route using `useSearchParams`:

1. Define defaults and parsing in `validateSearch`.
2. Read values with `Route.useSearch()`.
3. Update them with navigation `search`.
4. Add loader-relevant values to `loaderDeps`.

### Step 7: Migrate data loading

Move route-critical reads to loaders that coordinate with Query. Keep
mutations in Query hooks and invalidate precise query keys after success.

### Step 8: Replace navigation APIs

Replace imports and convert string navigation into typed objects.

### Step 9: Remove React Router

After all imports are migrated, remove `react-router-dom`. TypeScript will then
find any leftovers.

Two browser router providers should not manage the same application tree. If a
temporary dual-router migration boundary is unavoidable, it must be explicitly
isolated and documented.

### Step 10: Validate route behavior

Verify:

- Direct page refresh
- Browser back and forward
- Deep links
- Invalid params
- Invalid search values
- Logged-out protected links
- Post-login return URL
- Not-found routes
- Loader failures
- Unsaved form blocking

## 23. Common Failure Modes

### Manual modification of `routeTree.gen.ts`

`routeTree.gen.ts` must not be edited manually. Correct the route source file or
Vite configuration and regenerate the route tree.

### Stale generated route types

Restart `pnpm dev` or run:

```bash
pnpm build
```

### Child route does not render

Check that its parent component renders `<Outlet />`.

### Protected content renders before redirect

Move the auth decision from a component effect into parent `beforeLoad`.

### Search values have incorrect runtime types

URLs are untrusted serialized input. Parsing and fallback behavior should be
centralized in `validateSearch` rather than distributed across page
components.

### Loader does not rerun after a filter change

Put the relevant validated search values in `loaderDeps`.

### Navigation removes unrelated search state

Use a functional search update and spread the previous object.

### TypeScript rejects a valid browser path

Use the route's typed `to` path rather than its internal pathless route ID.
Regenerate the route tree after adding or renaming a route.

### Redirect loop

Check:

- Whether login's `beforeLoad` redirects authenticated users.
- Whether auth state and tokens agree.
- Whether `router.invalidate()` runs after auth changes.
- Whether the destination is itself protected.
- Whether the preserved redirect target is safe and valid.

### Duplicate loader and component requests

Use a shared Query options factory with `ensureQueryData` in the loader and
`useSuspenseQuery` in the component.

### Unrestricted use of `strict: false`

Data should be passed into shared components where practical. Non-strict router
hooks should be limited to components that must inspect an unknown route
branch.

## 24. Recommended Feature Organization

The following structure is recommended as route-backed features are added:

```text
src/
├── routes/app/
│   ├── _authenticated.users.tsx
│   ├── _authenticated.users.index.tsx
│   └── _authenticated.users.$userId.tsx
├── views/Users/
│   ├── UsersListPage.tsx
│   └── UserDetailsPage.tsx
└── services/users/
    ├── users.api.ts
    ├── users.queries.ts
    └── users.mutations.ts
```

Responsibility allocation:

- Route module: URL contract, guard, loader, route-level fallbacks, metadata.
- View: page composition and interaction.
- API module: HTTP requests.
- Query module: query options and keys.
- Mutation module: mutation hooks and invalidation.
